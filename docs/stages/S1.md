## S1 一句话目标
把**语料真源**落地:按 PRD §6 配方完成语料构建与多模态摄取,产出**真实数据合同级** `manifest.jsonl` 与 `errors.jsonl`,并做小样本 dry-run + 门禁校验后扩到全量。**本阶段交付的是语料真源与审计产物,不是任何检索效果。**

## S1 总边界(暂时不做)
不做 GT 冻结(S2)、不做查询集 / dev-test 分割(S2)、不做检索(S3/S6)、不做 baseline、不做 A→B、不做真实评测(S4/S5)、不做 QA 映射可行性探针(S2)、不做 CLIP、不做视频内容(ASR/抽帧/地点)、不做速度优化、**不产出任何"效果提升"结论**。

> 本阶段只回答两件事:① 语料是否按合同被构建出来;② 摄取结果是否足够诚实、可审计、可供后续 GT 与检索使用。  
> **`manifest.jsonl` / `errors.jsonl` 是本阶段真交付物,不是 S0 那种轻量评估视图,也不是临时中间产物。**

## 子阶段 S1.1 — 锁数据合同与不变量

**执行目标**:在动任何真实语料前,把 `manifest.jsonl` / `errors.jsonl` 的字段合同、来源披露与阶段不变量锁死,避免后续沿用 S0 的轻量评估视图或边做边改字段。

**步骤**
1. 以 ARCHITECTURE §5 `FileRecord` 为准,明确 `manifest.jsonl` 的必需字段与允许取值:
   - 基础字段:`id`、`rel_path`、`file_type`、`type_label`、`filename`、`lang`
   - 时间字段:`created_at`、`modified_at`、`created_epoch`、`modified_epoch`
   - 多模态字段:`exif`、`text`、`extract_status`
   - 追溯字段:`qa_links`、`source`、`synthetic_fields`
2. 明确 `errors.jsonl` 为**错误明细真源**,至少记录:
   - `rel_path`
   - `stage`
   - `error_msg`
   - 若实现需要可附加 `file_type` / `exception_type` / `source_kind`,但不得替代前三项
3. 进入 S1 前先锁关键不变量:
   - **所有日期范围匹配只认 `*_epoch` 物化字段**,不得留字符串比较后门
   - **photo / video / media 的 `text=null`**
   - **坏文件仍进 manifest**,只通过 `extract_status` 与 `errors.jsonl` 披露失败,不得静默丢弃
   - **合成字段必须进 `synthetic_fields`**,不得只在文档口头说明
   - **媒体永不作为任何查询正例,但仍保留在语料里**
   - **视频能力边界只到 `{time,type,filename}`**,不得补地点 / 语义能力
   - **照片不补语义 / 地点以外的新能力**,本期只认 EXIF 时间/地点/相册 + 文件名
4. 明确 `qa_links` 与 `source` 字段先按合同保留:
   - `qa_links` 为后续 S2 QA 物化 / 映射留钩子,本阶段可为空数组或仅对已物化样本填值
   - `source` 至少可区分 `public|standard_qa|synthetic` 与 `dataset`
5. 锁定候选池门禁口径:
   - `MIN_CANDIDATE_POOL` 为配置单一真源
   - 门禁按 **`type_label` × 可检索角色** 设计,不是只看总文件数
   - 不满足门禁时 `CorpusBuilder` **报错**,不得静默放行

**任务边界**:只锁合同与不变量,**不下载 / 不抽取 / 不建全量 manifest**。

**验收**
- `manifest.jsonl` / `errors.jsonl` 字段合同写清、无歧义。
- 关键不变量已显式写入本阶段真源,后续实现不得自由发挥。
- 已明确 **S1 真交付物是 manifest/errors 与审计信息,不是检索效果**。
- 已明确 `MIN_CANDIDATE_POOL` 门禁按 `type_label` 与可检索角色执行。

## 子阶段 S1.2 — 10–20 文件 dry-run(先小样本闭环)

**执行目标**:先用 **10–20 文件**的小样本跑通"来源 → 元数据合成 / 抽取 → manifest/errors 落盘"最小闭环,尽早暴露字段缺失、抽取异常与不变量破坏,而不是一上来跑全量。

**步骤**
1. 选取覆盖多类型的小样本(建议 10–20 文件),至少包含:
   - 照片
   - 视频
   - 扫描件 / 证件
   - 学习资料 / 文档
   - 笔记
   - 媒体
   - 至少 1 个故意保留的坏文件 / 抽取失败样本
2. 对样本执行最小语料构建:
   - 生成稳定 `id`
   - 写 `rel_path`(相对 `data/`)
   - 填 `file_type` / `type_label` / `filename` / `lang`
   - 物化 `created_at` / `modified_at` 与对应 `*_epoch`
3. 对不同类型执行对应摄取:
   - 文档 / 学习 / 笔记:抽文本进 `text`
   - 扫描件:OCR→`text`
   - 照片:抽 EXIF 时间 / 地点 / 相册到 `exif`,且 `text=null`
   - 视频:仅文件名 / 日期相关字段,`text=null`
   - 媒体:仅元数据,`text=null`
4. 对坏文件执行失败保留策略:
   - 记录到 manifest
   - 设置 `extract_status=ocr_failed|parse_failed|na`
   - 同步写一条 `errors.jsonl`
5. 核对合成披露:
   - 合成的日期 / 文件名 / EXIF 地点 / 相册等,逐项进 `synthetic_fields`
   - `source.kind` / `source.dataset` 与真实来源一致
6. 输出 dry-run 审计摘要:
   - 各类型样本数
   - `text!=null` / `text=null` 分布
   - `extract_status` 分布
   - `errors.jsonl` 条数与 stage 分布

**任务边界**:只做小样本闭环与字段审计,**不扩到全量,不讨论检索效果**。

**验收**
- 10–20 文件 dry-run 成功落出可读的 `manifest.jsonl` / `errors.jsonl` 样本。
- `rel_path` / `lang` / `created_at` / `modified_at` / `*_epoch` / `exif` / `text` / `extract_status` / `qa_links` / `source` / `synthetic_fields` 均已按合同出现。
- `photo/video/media` 的 `text` 均为 `null`。
- 坏文件**仍在 manifest 中**,并在 `errors.jsonl` 有对应错误明细。
- 未出现字符串日期替代 `*_epoch` 的情况。

## 子阶段 S1.3 — CorpusBuilder 门禁与候选池校验

**执行目标**:在 full run 前把 `CorpusBuilder` 的配方、候选池与能力边界门禁跑通,确保 S1 产物一开始就符合后续 S2/S3 需要,而不是先堆数据再返工。

**步骤**
1. 按 PRD §6 一期配方组织来源与目标占比,但本阶段关注点是**可检索角色是否成立**,不是视觉上凑够总量。
2. 以 PRD §6.1 与 ARCHITECTURE AD-5 为准,在构建期校验类型能力边界:
   - 照片:可提供 `{time,type,filename,location}`
   - 视频:只提供 `{time,type,filename}`
   - 证件 / 学习 / 笔记:可提供文本语义
   - 媒体:恒真负例角色,不赋予正例能力
3. 按 `type_label` 与可检索角色检查候选池:
   - 被 Q2 / Q6 类型过滤命中的 `type_label` 必须有足够候选数
   - 能进入正例池的类型,其支撑字段必须真的存在
   - 不得用媒体 / 视频去虚补地点或语义类候选
4. 执行 `MIN_CANDIDATE_POOL` 门禁:
   - 对每个被查 `type_label` 校验候选数是否达到配置阈值
   - 不足则直接报错并返回缺口,不得继续构建
5. 审核来源披露与数据组成:
   - `public` / `standard_qa` / `synthetic` 三类来源比例可被统计
   - 合成字段可被逐字段回溯
6. 形成门禁审计结果:
   - 各 `type_label` 候选池计数
   - 各文件类型按能力边界可支撑的查询角色概览
   - 门禁通过 / 失败原因列表

**任务边界**:只做配方与门禁校验,**不冻结 GT,不生成查询,不做任何 recall/precision 统计**。

**验收**
- `MIN_CANDIDATE_POOL` 作为硬门禁生效,不足即失败。
- 候选池口径清楚:按 `type_label` 与可检索角色,不是按总文件数糊弄。
- 视频未被赋予地点 / 语义能力,媒体未被赋予任何正例能力,照片未被补充语义能力。
- 可统计每类来源与合成字段披露情况。

## 子阶段 S1.4 — 全量语料构建与多模态摄取

**执行目标**:在 dry-run 与门禁通过后,按 PRD §6 配方扩到一期全量语料,产出可供后续阶段只读消费的 `manifest.jsonl` 与 `errors.jsonl`。

**步骤**
1. 按配方构建全量语料来源:
   - 照片 / 视频
   - 证件 / 扫描件
   - 学习资料
   - 个人笔记 / 文档
   - 媒体
2. 执行多模态摄取(FR-2):
   - 文档类抽文本
   - 扫描件 OCR
   - 照片抽 EXIF 元数据(时间 / 地点 / 相册)
   - 视频仅保留文件名 / 日期类信号
   - 媒体仅保留元数据
3. 统一写入 `FileRecord`:
   - `rel_path` 指向实际语料相对路径
   - `lang` 明确中 / 英
   - `created_at` / `modified_at` 与 `created_epoch` / `modified_epoch` 成对存在
   - `exif` 仅在适用类型出现有效内容
   - `text` 与文件类型能力边界一致
   - `extract_status` 反映真实抽取状态,不可全部假装 `ok`
4. 保留失败样本:
   - 抽取失败文件照常入 manifest
   - 同步落 `errors.jsonl`
   - 失败原因可回溯到 stage
5. 保留追溯字段:
   - `qa_links` 留后续 S2 接口
   - `source` 能回溯到数据集 / 合成来源
   - `synthetic_fields` 对每条记录逐项披露
6. 生成全量构建审计摘要:
   - 总文件数
   - 各 `file_type` / `type_label` 分布
   - `lang` 分布
   - `extract_status` 分布
   - `text=null` 与 `text!=null` 分布
   - `errors.jsonl` 计数与失败 stage 分布

**任务边界**:只交付全量语料真源与审计摘要,**不进入索引 / 检索 / 评测阶段**。

**验收**
- 全量 `manifest.jsonl` / `errors.jsonl` 已生成且结构稳定。
- `rel_path` / `lang` / `created_at` / `modified_at` / `*_epoch` / `exif` / `text` / `extract_status` / `qa_links` / `source` / `synthetic_fields` 全量可读、可审计。
- 坏文件被保留进 manifest,而不是被预处理静默删除。
- `photo/video/media` 全量保持 `text=null`。
- `errors.jsonl` 不是空壳;其条目能真实反映抽取失败。

## 子阶段 S1.5 — 阶段审计封口与交接

**执行目标**:确认 S1 只交付语料真源与审计产物,把后续 S2 需要的输入边界封口,避免带着"半成品字段"或"伪完成语料"进入 GT 阶段。

**步骤**
1. 做阶段内一致性核对:
   - `manifest` 字段与 ARCHITECTURE §5 合同一致
   - PRD §6 配方与实际分布可对上
   - PRD §6.1 资格矩阵边界未被破坏
2. 明确后续只读输入:
   - `manifest.jsonl`
   - `errors.jsonl`
   - 配置中的 `type_label` / `capability_matrix` / `MIN_CANDIDATE_POOL`
3. 输出 S1 阶段审计结论:
   - 哪些类型已具备后续 GT 派生所需信号
   - 哪些失败样本会影响后续 precision 压测,但不应被删
   - 哪些字段为 S2 保留接口(如 `qa_links`)
4. 明确 S1 不给出的东西:
   - 不给 baseline 数字
   - 不给 A→B
   - 不给 dev/test
   - 不给 QA 映射结论
   - 不给真实评测结论
5. 准备进入 S2 的最小输入清单,但**不自动进入 S2**。

**任务边界**:只做 S1 封口与交接,**不提前实现 S2 内容**。

**验收**
- 已明确 S1 真正完成的是语料真源与审计产物,不是检索能力证明。
- 后续阶段只读输入清楚,无字段漂移空间。
- S1 与 S2 的边界被显式写清,不会自动越界推进。
