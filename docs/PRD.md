## 0. 一句话定义

在一个**像真实个人 NAS(影像为主、中文为主)的仿真文件语料**上,用"**评估优先**"的方法,量化验证:**真实的"找文件"需求大量是非语义的,因此混合检索(元数据过滤 + BM25 + 向量 + 重排)优于纯语义检索(优幅由实测决定,不预设)。**

> 本项目交付的不是一个上线产品,而是**一个评估驱动的多模态检索质量工程切片**。这一点贯穿全文,任何描述都不得越界为"已落地产品"。

---

## 1. 背景与问题

### 1.1 产品背景
面向**个人用户**的 NAS(本地网络存储)。个人用户把各类文件存入本地 NAS,在 NAS 本地按多种检索策略(元数据 / 语义 / 混合)查询(远程接入是产品形态,**本期不涉及其传输 / 权限实现**)。个人用户存入的内容按**存储量**排序:**① 影像(照片 / 视频)② 证件 / 凭证 ③ 学习 / 资料 ④ 个人笔记 ⑤ 媒体**(评估时照片与视频分列,见 §6);母语为中文(约 70% 文件),夹带英文资料(约 30%)。数据具有"**本地化、不出门**"的天然定位。

### 1.2 真实痛点(来自真实使用 / 用户老师反馈)
- **痛点 A:检索效果不好**(找不准、找不到)。
- **痛点 B:检索速度不够快**(本期**仅分段测量**,优化留二期)。

### 1.3 为什么是这个切片
一期**只解决痛点 A + 建立评估体系**;痛点 B(速度 / 端侧)留二期。原因是因果顺序:**先有评估与诊断,才知道该优化什么**;先优化再补评估 = 得到说不清来源的漂亮数字。

## 2. 目标与非目标

### 2.1 核心命题(待量化验证的假设)
个人找文件的需求**大量是非语义的**——按时间、类型、文件名、拍摄地点查找(例:"去年七月在三亚拍的照片""找一张发票""名字里带 Q3 的那个表")。**纯语义向量检索在这类查询上系统性失效。** 命题分两层验证:
- **第一层(非语义基础)**:时间 / 类型 / 文件名等**元数据过滤** > 纯向量检索。
- **第二层(有内容文本处)**:**完整混合**(元数据过滤 + BM25 关键词 + 向量 + 重排) > 纯向量检索。

两层的提升幅度均**由实测决定,不预设目标数字**。

### 2.2 目标(可度量)
- **G1**:建立**可复现、Ground Truth(GT)锁死、按查询类别 × 文件类型拆分**的评测体系。
- **G2**:跑出诚实 baseline(纯语义),并在**同一套评测**上证明混合检索带来**可测量的提升**。*(提升数值由真实测量决定,本文不预设目标数字——预设漂亮数字本身违反诚实原则。)*
- **G3**:对检索全链路**分段计时**(分段清单见 §7),定位耗时瓶颈,为二期(速度优化)提供基线证据。

### 2.3 非目标(本期明确不做)
- ❌ 远程传输 / 权限 / 多租户 / 产品化 / 上线部署
- ❌ 重型图文跨模态融合:**CLIP 视觉语义检索为二期优化项,本期不实现**;一期仅基于 EXIF / 元数据 / 文本
- ❌ 视频内容理解(ASR / 抽帧)、视频拍摄地点 / 内容检索(留二期)—— **v1 视频仅按文件名 / 日期可检索,无 EXIF 地点**
- ❌ 音频转写(留二期)
- ❌ 速度与端侧优化(二期)
- ❌ 回答生成质量(faithfulness 仅作收尾**可选小验证**,且连主线:更好检索 = 抑制幻觉)

## 3. 目标用户与使用场景

### 3.1 用户画像
个人 NAS 用户,**单用户、无权限管理,不涉及多租户 / 共享场景**。母语中文(约 70% 文件与查询为中文),夹带英文资料(约 30%,如论文、英文电子书)。文件横跨数年、类型混杂、命名随意(部分文件名隐晦如 `IMG_2307.jpg`、`扫描件_04.pdf`)。核心诉求:**在大量杂乱文件里快速找到那一个**。

### 3.2 关键查询场景(用户故事)

| # | 查询类别 | 用户原话(例) | 主要检索信号 | 含非语义信号? |
|---|---|---|---|---|
| Q1 | 时间 | "去年 Q1 拍的照片" | 日期 | ✅ |
| Q2 | 类型 | "找一张发票""所有合同" | 文件类型 | ✅ |
| Q3 | 文件名 | "名字带 Q3 的那个表" | 文件名片段 | ✅ |
| Q4 | 地点(照片) | "三亚那次旅游的照片" | EXIF 地点(仅照片) | ✅ |
| Q5 | 语义 | "关于深度学习的那篇资料" | 内容语义 | ❌(纯语义) |
| Q6 | 混合 | "去年的年度预算 PDF""关于深度学习、2024 年后的论文" | 时间 + 类型 + 关键词 / 语义 | ✅(含语义) |

> 列「含非语义信号?」:✅ = 含时间 / 类型 / 文件名 / 地点等非语义信号;❌ = 纯语义、无非语义信号。**6 类查询里 Q1–Q4 与 Q6 共 5 类含非语义信号,仅 Q5 是纯语义。**

### 3.3 关键洞察
上表 6 类查询里 **5 类含非语义信号**(仅 Q5 纯语义),且影像(存储最大类,~40%)**缺内容理解**(本期不做 CLIP / ASR)——这直接推导出 2.1 的命题:**纯语义不够,必须混合检索**。

> 影像中**照片**有 EXIF(时间 / 地点 / 相册)+ 文件名可供非语义检索,但无内容文本;**视频**在 v1 仅文件名 / 日期、内容理解留二期,故对"地点 / 语义 / 内容"类查询是**真负例**。这反映 v1 的技术边界,同时客观印证命题:内容信号缺位 → 只能靠元数据 / 文件名 / 类型。

## 5. 功能需求(FR)

- **FR-1 语料构建与元数据**:汇集 / 合成仿真文件,为每个文件挂 `文件名 / 创建-修改日期 / 类型 / 来源 / (照片)EXIF 时间·地点·相册 / (视频)仅文件名·日期`,产出 `manifest.jsonl`。合成字段须标注来源。
- **FR-2 多模态摄取**:① 文档(PDF/Word/txt/md)抽文本;② 扫描件 OCR → 文本;③ 照片抽 EXIF 元数据(时间/地点/相册)+ 文件名;④ 视频仅取文件名/日期(本期不抽视觉特征、不做 ASR/抽帧)。
- **FR-3 索引**:统一索引层,支持**向量索引 + 元数据过滤 + 词法(BM25)**三种检索信号。
- **FR-4 检索 · 基线**:纯语义(向量)检索,作为对照靶子。
- **FR-5 检索 · 混合**:元数据过滤 + BM25 + 向量召回 + 重排(rerank)。
- **FR-6 评测集与 GT 管理**:**60–80 条**按 Q1–Q6 **均衡分布**(各类 ≥8 条)的查询;GT 客观推导(类型标签 + 合成元数据 + 数据集自带问答)为主、少量手标;**跑检索前锁死**(连同样本计数表与「文件类型 × 查询类别」资格矩阵一起,见 §6.1);**dev / test 分割**(整体 80/20,每类按比例,保证 test 每类 ≥2)。GT 设计完全独立于任何系统输出。
- **FR-7 评估与诊断**:计算 recall@k / precision@k / nDCG / MRR,**两个维度报告**——(A)按查询类别 Q1–Q6 聚合;(B)结合 §6.1 资格矩阵**按文件类型出 6×6 细分(TP/FP)**;Q6 拆「纯非语义部分 vs 含文本部分」分别报告。全链路**分段计时**(清单见 §7);产出可读的诊断报告、"A→B"对比及**失败案例详列**。

## 6. 数据 / 语料规格(一期配方)

**总量 ~700 文件,中文 70% / 英文 30%**,按存储量排序:

| 大类(按量) | 占比 / 数量 | 处理 | 检索信号 | GT 来源 | 数据来源 |
|---|---|---|---|---|---|
| ① 照片 | ~34% / ~240 | EXIF + 文件名索引(CLIP 二期,本期不做) | 时间/地点/相册/文件名 | 合成元数据可算 | 公开图片 + 合成 EXIF |
| ② 视频 | ~6% / ~40 | 仅文件名/日期(无 EXIF 地点;ASR/抽帧二期) | 时间/文件名 | 元数据可算;对内容/地点查询为真负例 | 公开视频/占位 + 合成元数据 |
| ③ 证件/凭证(扫描) | ~24% / ~165 | OCR→文本 | 类型/关键词/时间 | 类型标签 + 元数据 | 票据数据集 + 合成中文票据/账单/合同 |
| ④ 学习/资料 | ~18% / ~130 | 抽文本 | 语义/类型/时间 | 数据集自带问答 / 合成 | arXiv(英)+ 中文文章/电子书 |
| ⑤ 个人笔记/文档 | ~12% / ~85 | 抽文本 | 语义/文件名 | 数据集问答 + 少量手标 | 合成 + 公开文本 |
| ⑥ 媒体(音乐/录音) | ~6% / ~40 | 仅元数据(纯噪声) | 不作任何查询正例 | 对全部查询恒为真负例 | 公开元数据 / 合成 |

> 占比为**近似值**(四舍五入至整数 %),精确值见数量列(如照片 240/700 = 34.3% ≈ 34%;视频与媒体各 5.7% ≈ 6%)。

**语义 GT 数据集**:中文 DuReader / CMRC;英文 DocVQA / SQuAD —— 一期把这些集的"来源段落/版面"在 S1 **物化为独立文件**(1 问题 → 1 文件)后作 file-level GT,**半自动映射 + 抽检手标**(S2 前以 10 条探针验证映射可行性,不夸大为"全自动免手标")。
**数据来源类别(诚实披露)**:公开数据集(RVL-CDIP / arXiv / DocVQA 等)+ 标准问答集(DuReader / CMRC / SQuAD)+ **合成**(中文票据 / 账单 / 合同 / 部分笔记;中文资料缺标准问答处用模板 / 规则生成 GT)。**所有合成内容在 `manifest` 标注**,与 §12 对应。
**配方反推原则**:每个被查类型须有足够候选池(如类型查询每类 ≥30),否则查询无区分度。

> 注:照片 + 视频合计仍是存储第一大类(~40% / 280),忠实于真实个人 NAS;但**检索支柱是照片**(EXIF 时间/地点/相册 + 文件名;照片与视频均不支持视觉 / 内容语义,见 §6.1),视频只挂文件名 / 日期。

## 6.1 资格矩阵(文件类型 × 查询类别 · 与 GT 一起锁死)

> 语料按"存储占比"分类,评估按"可检索角色"分类——**两者不是一回事**。下表预先声明每个文件类型对每类查询是「✅ 可作正例」还是「➖ 仅真负例(永不是正确答案)」,**与 GT 一起在跑检索前锁死**(见 FR-6),让每个按类别的指标都能一句话归因。

| 文件类型 | Q1 时间 | Q2 类型 | Q3 文件名 | Q4 地点 | Q5 语义 | Q6 混合 |
|---|---|---|---|---|---|---|
| 照片 | ✅ | ✅ | ✅ | ✅ | ➖ | ✅ |
| 视频 | ✅ | ✅ | ✅ | ➖ | ➖ | ✅(仅时间/类型/文件名,不含地点/语义) |
| 证件/凭证 | ✅ | ✅ | ✅ | ➖ | ✅ | ✅ |
| 学习/资料 | ✅ | ✅ | ✅ | ➖ | ✅ | ✅ |
| 笔记 | ✅ | ✅ | ✅ | ➖ | ✅ | ✅ |
| 媒体 | ➖ | ➖ | ➖ | ➖ | ➖ | ➖ |

- ✅ = 可作此类查询的正确答案(进 GT 正例);➖ = 仅作真负例,用于压测 precision / 鲁棒性。
- **TP/FP 归因规则**:对每条(查询,文件类型),按本矩阵判定——仅当查询中**该文件类型有资格的分量**匹配时计 TP;命中了该类型**无资格的分量**(如视频被"地点 / 语义"分量召回、媒体被任何查询召回)计 FP;无资格一律不进正例池。因此 Q6 对视频只认"时间 / 类型 / 文件名"分量。**实现上**:矩阵在 GT 派生期就按分量资格筛定正例集,评估期归因退化为**成员判定**(命中 ∈ 冻结正例 = TP,否则 = FP),无需检索器上报来源(机制见 ARCHITECTURE §3 AD-5)。
- **视频** 退出 Q4(地点)/ Q5(语义)正例:v1 无 EXIF 地点、无内容理解。强行算正例会把 Q4 recall 天花板钉在数据缺陷上、污染最干净的非语义论据;贴合成 GPS 则是后向填补数据,违反诚实纪律。两者皆不可。
- **照片 Q5 = ➖** 仅指本期不做 CLIP 视觉语义;照片靠 EXIF / 文件名的非语义信号已在 Q1–Q4 / Q6 覆盖,不依赖内容文本。
- **媒体** 全行 ➖:评测集**因此不构造指向媒体的查询**,它专职检验"搜文档 / 证件时是否误捞无关音频"。
- 这张矩阵本身是面试展示物——"数字怎么来的"最硬的证据。

## 7. 评估方案与验收标准

- **指标与维度**:recall@k、precision@k、nDCG@k、MRR(k ∈ {1,3,5,10},主指标 k=5),**两个维度报告**:(A)按查询类别 Q1–Q6 聚合;(B)结合 §6.1 资格矩阵**按文件类型出 6×6 细分(TP/FP)**(例:Q1 对照片 recall vs 对视频 recall 分别看)。
- **速度 · 分段计时**:分段清单 = 语料加载 / OCR / embedding / 向量检索 / 元数据过滤 / BM25 / 重排 / 排序;工具 = Python `time` + `cProfile`;区分"检索本身慢"与"网络 / 传输慢"。**本期只测不优化。**
- **归因纪律**:每个按类别的 recall / precision,依 §6.1 资格矩阵可一句话归因到「检索能力」还是「数据建模约束」;视频 / 媒体等真负例类专用于压测 precision(搜文档是否误捞),不进任何 recall 正例。
- **诚实纪律**:GT 跑前锁死、不回改;调参只在 dev、test 只报一次;合成数据如实披露;数字附 baseline + 测试集规模 + **仍失败的案例**;**杜绝"又圆又大"的裸数字**(如 62%→94% 无方法说明)。
- **失败案例呈现**:(1)按查询类型统计失败率(如"X% 的 Q2 正确文件排名 > k");(2)按文件类型分析失效原因;(3)Top-3 最悬殊失败查询清单。
- **验收(Definition of Done)**:
  - DoD-1:评测集存在、GT 锁死、dev/test 分割、构造方法文档化。
  - DoD-2:纯语义 baseline 各类别指标 + 分段计时已产出。
  - DoD-3:混合检索在同一评测集、**同指标体系**复测,产出按类别拆分的"A→B"对比与失败分析。
  - DoD-4:诊断结论成文(哪类查询失败 / 时间耗在哪段)。
  - DoD-5:诚实性产出齐全——失败案例详列 + 合成数据比例 + 能力边界声明。

## 8. 非功能需求(NFR)

- **本地化**:全链路可纯本地运行,数据不出门(契合 NAS 定位);不依赖闭源云 API。
- **可复现**:固定随机种子、固定语料 manifest;同输入同结果。
- **弱算力友好**:CPU 可跑(NAS 无 GPU 是真实场景)。**当前未优化,耗时仅作测量,是二期速度诊断的基线,非本期性能目标。**
- **可替换**:向量库 / embedding / reranker 为可替换零件(改配置即可 A/B)。

## 9. 技术选型(含理由)

| 组件 | 选型 | 为什么(及为什么不选别的) |
|---|---|---|
| 编排 | **LlamaIndex** | 核心是检索非编排;检索优先、索引策略丰富。LangChain 偏 Agent,留作他用。 |
| 向量库 | **Chroma(出数)+ QdrantStore 证明可换(真迁移可选)** | Chroma 出全部评测数字;另写 QdrantStore 适配 + 契约测试证明"换库=改配置",700 文件下不真迁移;Qdrant(纯本地 + 原生混合检索 + 过滤,契合"数据不出门")。FAISS 是检索库非数据库,缺持久化/元数据过滤/增删改;不用 Milvus(过度工程)/ Pinecone(闭源云)。 |
| Embedding | **bge-m3** | 硬约束(中英混 + 本地 + 长上下文)先淘汰 OpenAI(闭源云)/ 短上下文小模型;bge-m3 满足需求;**一期仅用 dense**,sparse/colbert 列二期扩展(本期词法由独立 BM25 承担,不宣称已用多模式)。**embedding 品牌非项目核心(核心是评估体系)**,一期定基线,有评测后可 A/B 替换。 |
| 词法 | **BM25** | 文件名/关键词类查询的关键信号。 |
| 重排 | **bge-reranker-v2-m3** | 与 bge-m3 同源,配套干净。 |
| OCR | 扫描件文本化(决策链 PaddleOCR→RapidOCR→Tesseract,装包性 S0/S1 验证) | 让扫描件可检索,并提供"纯文本抽取失效→OCR 修复"的诊断线。 |
| 视觉 | **CLIP:本期不用 → 二期** | 照片 EXIF + 文件名已覆盖 Q1–Q4 必要信号;通用 CLIP 对中文家庭场景语义有限、CPU 开销大。若实测发现 Q4/Q5 瓶颈确因视觉信息缺位,二期重评。 |
| 环境 | **conda + Python 3.12** | 规避系统 Python 3.14 装不上 ML 栈;conda 管原生依赖更稳。 |

## 12. 诚实性声明与已知局限

- 语料为**公开数据 + 合成**拼装的**仿真个人 NAS**,真实用户数据不可得;部分类型(中文票据/账单/合同等)为模板合成,均在 `manifest` 标注。
- 评测查询为**人工构造(半合成)**,但 GT 尽量锚定客观标签与数据集自带问答,并在跑检索前锁死。
- 本项目的"**多模态**"本质是多种内容格式(文档 / 图像 / 视频)各自提取对应信号(文本 / EXIF / OCR),**不做跨模态融合(如 CLIP 视觉-语言对齐)**,不夸大为重型多模态检索。
- 本期**不交付上线产品**,不声称权限 / 并发 / 系统集成等未实际完成的工程能力。
- v1 **视频仅按文件名 / 日期可检索**(无 EXIF 地点、内容理解留二期);**媒体为纯噪声类**(不作任何查询正例,仅压测 precision)。二者对"内容 / 地点 / 语义"查询均为**真负例**,已在 §6.1 资格矩阵中随 GT 锁死。

## 附录 A. 对外叙事(面试定位)

- **一句话价值**:"在自己几千个横跨多年、杂乱的文件里**精准找到**去年那张报销单——纯语义做不到,因为这类需求大多按时间 / 类型 / 文件名 / 地点找,混合检索可量化证明显著更优。"
- **差异化**:不靠炫技技术栈(那是刻意的),靠**评估那一圈 + 真实约束**;能拿出锁死评测集、按类别 × 类型拆分的 A→B、以及仍失败的案例。
- **预判追问**:① "数字怎么来的"→ 出示评测集与方法;② "NAS 是仿真的算真实吗"→ 文件可仿,**查询分布与方法是真的、可迁移**,且有真实文件小验证;③ "做没做幻觉抑制"→ 更好检索就是最高杠杆的幻觉抑制,并有前后对比验证;④ "影像是最大类,视频能按内容 / 地点找吗"→ v1 视频只按文件名 / 日期找、内容 / 地点留二期,这是**主动声明的能力边界**,也印证命题(内容信号缺位 → 只能靠元数据 / 类型)。
